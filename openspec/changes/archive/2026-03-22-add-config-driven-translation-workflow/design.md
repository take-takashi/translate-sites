## 背景

現状のリポジトリには、翻訳対象サイトを継続管理するための正式な OpenSpec change が存在しません。試験的に `mise` の `tasks` ページを `sites/mise.yml`、`data/raw/mise/tasks/index.html`、`data/ja/mise/tasks/index.md`、`data/state/mise.json` として保存し、設定ファイル、raw、ja、state の4点セットが基本単位になることを確認しました。

このプロジェクトの目的は、1回きりの実行ではなく、対象サイトの原文取得、翻訳結果、状態管理を継続して扱えるようにすることです。そのため、CLI の有無より先にファイル配置と責務分離を固める必要があります。

## 目標 / 非目標

**目標:**
- サイト設定、raw 保存、ja 保存、state 追跡の責務を明確に分ける
- 同じ raw から再翻訳や差し替えができる保存構造を定義する
- `mise` のような技術ドキュメントサイトを迷わず管理できる最小ワークフローを定義する

**非目標:**
- サイト全体を自動クロールする仕組み
- 高度な翻訳品質調整や用語集管理
- エンドユーザー向けの完成済みCLI製品化
- GitHub Pages 向けの公開生成までをこの change に含めること

## 設計判断

### 1. 設定ファイルを起点にサイト単位で管理する
各サイトは `sites/<name>.yml` のような設定ファイルで定義し、対象URL、許可ドメイン、保存先をそこに集約します。これにより、どのページをどのサイトとして管理しているかをコード外で明示できます。

代替案として URL を都度指定する運用もありますが、対象の再取得や更新追跡時に管理が分散するため採用しません。

### 2. raw と ja を物理的に分離する
取得した原文は `data/raw/<site>/...`、翻訳済み成果物は `data/ja/<site>/...` に分けて保存します。こうすることで、raw を保持したまま翻訳だけ差し替える、または翻訳方針を見直して再生成するといった運用が可能になります。

代替案として raw を持たず翻訳済みだけ残す方法もありますが、再取得や比較、抽出改善時の再利用性が低いため採用しません。

### 3. state は site 単位の JSON で持つ
状態管理は `data/state/<site>.json` に集約し、ページURL、raw パス、ja パス、取得時刻、翻訳時刻、ハッシュ、メモなどを持たせます。MVP では人が読めることと追記しやすいことを優先し、JSON を採用します。

代替案として frontmatter だけに状態を分散させる方法もありますが、サイト全体の進捗確認や更新判定がしづらくなるため採用しません。

### 4. MVP の翻訳入力は raw から明示的に決める
MVP では、raw からそのまま全文を機械的に処理するのではなく、「どの部分を翻訳対象とするか」を人間または後続の抽出処理で決められる余地を残します。これにより、サイドバーやナビゲーションを無理に翻訳対象へ含めず、本文中心の運用を維持できます。

代替案として raw HTML 全体を即時翻訳する方法もありますが、ノイズ混入が大きく、管理対象としての品質を保ちづらいため採用しません。

## リスク / トレードオフ

- [設定ファイルが肥大化する] → URL追加ルールと最小必須項目を先に決め、MVPでは項目数を増やしすぎない
- [raw と ja の対応が崩れる] → state に raw パスと ja パスを必須で記録する
- [本文抽出ルールが曖昧になる] → MVPでは sample ページを使って翻訳対象の粒度を明示し、後続 change で抽出ルールを固める
- [更新追従の設計が先送りされる] → state に取得時刻と source hash を持たせ、将来の差分判定に備える
