## Context

現状のリポジトリには、プロジェクト方針を定義したREADMEのみがあり、Webページ取得や翻訳の実装は存在しません。MVPでは、まず単一URLを対象にしたCLIに限定し、クロールや差分更新のような拡張機能は扱いません。

HTMLをそのまま翻訳するとナビゲーションやフッターなどのノイズが混ざりやすいため、本文抽出を独立した工程として扱う必要があります。また、翻訳前にMarkdownへ変換しておくことで、見出しやリンクなどの文書構造を保持したまま後編集しやすい出力を維持できます。

最初の対象サイトには `mise-en-place` のドキュメントサイトを採用します。理由は、技術文書として典型的な見出し構造、内部リンク、コードブロックを備えており、本文抽出とMarkdown構造維持の検証に向いているためです。

## Goals / Non-Goals

**Goals:**
- 単一URLを入力とするCLIの責務を明確にする
- 取得、抽出、変換、翻訳、保存を分離したパイプラインとして設計する
- 初期実装で必要な品質基準と失敗時の扱いを決める
- `mise-en-place` の代表ページを通じて、技術ドキュメントに対するMVP品質を確認できるようにする

**Non-Goals:**
- サイト全体クロールや再帰的リンク追跡
- 認証付きページや高度なJavaScriptアプリへの完全対応
- 翻訳品質の完全自動保証や用語集管理

## Decisions

### 1. 単一責務のパイプライン構成を採用する
CLI は「入力受付」と「結果の報告」に絞り、内部処理は `fetch -> extract -> convert -> translate -> save` の段階に分離します。これにより、将来 `fetch` を `Playwright` ベースへ差し替える、あるいは翻訳バックエンドを変更する場合でも影響範囲を限定できます。

代替案として、取得後のHTMLを一括で翻訳してからMarkdown化する方法もありますが、文書構造が壊れやすく、変換後の品質が不安定になるため採用しません。

### 2. 翻訳対象は本文由来のMarkdownに限定する
翻訳は、本文抽出済みのコンテンツをMarkdownへ変換した後に実行します。これにより、翻訳対象からノイズを減らし、見出しやリンク、コードブロックの維持ルールを適用しやすくなります。

代替案として本文抽出前のHTML断片を翻訳する方法もありますが、DOM構造や属性が崩れやすく、後段の整形コストが高くなります。

### 3. 保存時に追跡可能なメタ情報を付与する
出力されたMarkdownには、少なくとも元URLを含められる形式を前提にします。初期実装では frontmatter もしくはファイル先頭の固定メタ情報ブロックのいずれかを採用できるようにし、仕様では「元URLが出力に残ること」を必須とします。

代替案として本文だけを保存する方法もありますが、再取得や手修正時の追跡性が落ちるため採用しません。

### 4. `mise-en-place` の代表URLを受け入れ確認に使う
初期検証では、少なくとも `https://mise.jdx.dev/`、`https://mise.jdx.dev/getting-started.html`、`https://mise.jdx.dev/installing-mise.html` のような代表ページを確認対象に含めます。これにより、トップページに近い構造、通常のドキュメントページ、コードブロックを多く含む導入ページをそれぞれ評価できます。

代替案として任意URLだけで確認する方法もありますが、抽出品質や構造維持の確認観点がぶれやすいため、最初のサンプル群は固定します。

## Risks / Trade-offs

- [本文抽出精度] → 汎用抽出ではサイトによってノイズが残る可能性があるため、MVPでは「本文中心であること」を満たしつつ、抽出器を差し替え可能な構成にする
- [翻訳によるMarkdown破損] → 翻訳前後で見出し・リンク・コードブロックを維持する検証をタスクに含める
- [動的ページへの非対応] → 初期実装は静的取得を前提にし、将来 `fetch` の差し替えで対応できるよう段階分離を保つ
- [外部API依存] → 翻訳処理は抽象化し、失敗時のエラーハンドリングをCLIから明示できるようにする
- [`mise-en-place` への過剰最適化] → MVPでは `mise` を受け入れ確認に使うが、抽出器や変換器は他サイトにも適用できる汎用構成を維持する
