## 背景

正式仕様としては `openspec/specs/` に 4 capability が定義済みですが、実装はまだ存在せず、`sites/mise.yml` と `data/` 配下の sample は人手で作った状態です。この change では、仕様をいきなり大規模機能へ広げるのではなく、設定の読み込み、保存先 path の決定、state 更新の3つを共通化して、今の sample を同じルールで再生成できる最低限の実装を作ります。

## 目標 / 非目標

**目標:**
- `sites/*.yml` の最小必須項目を検証できるようにする
- raw / ja / state の保存先 path を共通ルールで決定できるようにする
- state JSON の更新処理を共通化し、sample をその処理で扱えるようにする

**非目標:**
- サイト全体クロール
- 高度な本文抽出や自動翻訳品質向上
- 外部サービス連携の本格実装
- GitHub Pages 向け公開処理

## 設計判断

### 1. bootstrap 実装は Node.js 標準機能を優先する
最初の実装では、設定検証、path 生成、JSON 更新といった基盤処理が中心です。外部依存を増やす前に、Node.js 標準機能で完結する範囲を優先します。

代替案として最初から YAML/validation ライブラリを厚く入れる方法もありますが、MVP 段階では責務が小さく、追加依存の判断を急ぐ必要はありません。

### 2. sample を正解データ兼テストフィクスチャとして使う
`mise` の sample は、仕様の例であると同時に、実装の確認対象として扱います。これにより、実装が current workflow を壊していないかを小さく確認できます。

代替案として別の fixture を用意する方法もありますが、いま必要なのは既存 sample を再現可能にすることなので採用しません。

### 3. path ルールは設定値と URL から一意に決まるようにする
`raw_dir`, `ja_dir`, `state_file` は設定ファイルで明示し、ページごとの path は URL から機械的に決められるようにします。これにより、raw / ja / state の対応関係を人手で揃えなくて済みます。

代替案として state 側だけを正本にして各 path を都度書く方法もありますが、保存規則が分散するため採用しません。

### 4. state 更新は全置換ではなく page 単位で扱う
site state は JSON 全体を毎回手で編集するのではなく、ページ単位レコードを更新する前提にします。これにより、ページ追加時に既存レコードを壊しにくくなります。

代替案として state を都度丸ごと書き換える運用もありますが、更新箇所の追跡性が低く、今後の差分更新にも不利です。

## リスク / トレードオフ

- [YAML 解析手段が弱い] → 最初は書式を厳しく限定し、複雑な YAML を許さない
- [sample 依存が強すぎる] → 実装の責務は sample 固有ロジックにせず、sample は確認対象として扱う
- [state 更新の粒度が曖昧になる] → page record の必須項目を先に固定する
- [path ルールの後方互換が崩れる] → 既存 sample の path を壊さない形で実装する
